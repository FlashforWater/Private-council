# Private Council MVP Implementation Brief

This brief is for a coding agent implementing the first usable version of Private Council.

Read this together with:

- `PRIVATE_COUNCIL_PRODUCT_CONSTITUTION.md`

The constitution explains the long-term philosophy and system design. This brief narrows the first implementation to a manageable vertical slice.

## 1. MVP Goal

Build a web app that lets a user create one personal decision session, participate in a structured AI council discussion, watch a Decision Canvas evolve, make a final decision, and schedule a future retrospective.

The MVP should prove this product difference:

> Private Council is not multi-agent chat. It is a structured decision session that produces a traceable decision record and a future review plan.

## 2. Product Scope

### 2.1 In Scope

Implement one end-to-end decision flow:

1. User creates a decision session.
2. System collects basic context.
3. A fixed council of roles participates.
4. The session moves through a small state machine.
5. Agent outputs are mocked or generated by one model adapter.
6. Decision Canvas updates from structured outputs.
7. User can ask, challenge, clarify, decide, and schedule review.
8. Final Decision Record is generated.
9. Session is stored locally or in a simple database.

### 2.2 Out of Scope for MVP

Do not implement these yet:

- Full multi-provider model routing
- True long-term reliability learning
- Complex Brier score dashboards
- Full privacy management UI
- Real user accounts and teams
- Real avatar animation or Live2D
- Full web search citation pipeline
- Calendar integration
- Payment or billing
- Mobile app
- Clinical, legal, investment, or medical decision support

The code should leave clean extension points for these, but not implement them prematurely.

## 3. Recommended Tech Stack

If starting from scratch:

- Frontend: Next.js + React + TypeScript
- Styling: Tailwind CSS or existing project styling
- State: React state or Zustand for MVP
- Backend: Next.js route handlers or a lightweight Node API
- Storage: SQLite or JSON file storage for MVP
- Schema validation: Zod
- LLM adapter: interface-first, with a mock adapter as default

Do not block MVP on real API keys. The app should run in mock mode.

## 4. Core User Flow

### 4.1 Create Session

User enters:

- Decision question
- Short background
- Current leaning
- Time horizon
- Emotional state
- Optional constraints

Example:

```text
Should I quit my job to work on my AI product?
```

### 4.2 Framing

Chair refines the question:

- Original question
- Refined decision question
- Decision type
- Known facts
- Unknowns
- Constraints

User confirms or edits.

### 4.3 Independent Views

Council roles produce independent initial views.

MVP roles:

- Chair
- Strategist
- Skeptic
- Operator
- Researcher
- User Advocate

Reflector appears only at review stage, not during normal MVP session.

### 4.4 Challenge

Skeptic challenges the strongest option. User can press:

- Challenge this
- Ask role
- Clarify context
- Continue

### 4.5 Evaluation

System produces options and criteria.

User can adjust criterion weights.

Minimum criteria:

- Long-term alignment
- Expected upside
- Execution cost
- Risk
- Reversibility
- Emotional cost
- Information sufficiency

### 4.6 Recommendation

Chair summarizes:

- Recommended option
- Rationale
- Confidence
- Key assumptions
- Main objections
- Minimum next action
- Review timing

### 4.7 Human Decision

User chooses:

- Accept recommendation
- Modify recommendation
- Choose another option
- Defer decision
- Reject all

User can enter rationale.

### 4.8 Commitment and Review

User sets:

- Next action
- Success criteria
- Review date or review trigger

The session becomes a decision record.

## 5. MVP Session State Machine

Use a clear state machine. Do not rely on loose chat state.

```ts
type SessionPhase =
  | "created"
  | "framing"
  | "context_collection"
  | "independent_views"
  | "challenge"
  | "evaluation"
  | "recommendation"
  | "human_decision"
  | "commitment"
  | "scheduled_review"
  | "retrospective"
  | "closed";
```

Allowed MVP transitions:

```text
created -> framing
framing -> context_collection
context_collection -> independent_views
independent_views -> challenge
challenge -> evaluation
evaluation -> recommendation
recommendation -> human_decision
human_decision -> commitment
commitment -> scheduled_review
scheduled_review -> retrospective
retrospective -> closed
```

Allow user to move backward from:

- framing -> context_collection
- recommendation -> challenge
- human_decision -> evaluation

## 6. Core Data Model

Use the constitution as the long-term model, but implement this MVP subset.

### 6.1 Session

```ts
type CouncilSession = {
  id: string;
  title: string;
  status: "active" | "decided" | "scheduled_for_review" | "reviewed";
  currentPhase: SessionPhase;
  createdAt: string;
  updatedAt: string;
  canvas: DecisionCanvas;
  transcript: Message[];
  agentRuns: AgentRun[];
};
```

### 6.2 Decision Canvas

```ts
type DecisionCanvas = {
  id: string;
  problem: ProblemFrame;
  context: DecisionContext;
  emotionalContext?: EmotionSnapshot;
  options: Option[];
  criteria: Criterion[];
  claims: Claim[];
  evidence: Evidence[];
  assumptions: Assumption[];
  objections: Objection[];
  risks: Risk[];
  predictions: Prediction[];
  recommendation?: CouncilRecommendation;
  humanDecision?: Decision;
  actionCommitments: ActionCommitment[];
  reviewPlan?: ReviewPlan;
};
```

### 6.3 Minimum Structured Agent Output

Every agent run should produce this shape:

```ts
type StructuredAgentOutput = {
  summary: string;
  claims: ClaimDraft[];
  assumptions: AssumptionDraft[];
  risks: RiskDraft[];
  questionsForUser: string[];
  recommendedActions: string[];
  confidence?: "low" | "medium" | "high";
};
```

## 7. Council Roles for MVP

### 7.1 Chair

Purpose:

- Keep the session structured.
- Refine the decision question.
- Summarize disagreement.
- Generate recommendation.

Must not:

- Pretend to make the final decision.
- Hide important disagreement.

### 7.2 Strategist

Purpose:

- Evaluate long-term alignment.
- Identify opportunity cost.
- Compare direction-level tradeoffs.

### 7.3 Skeptic

Purpose:

- Attack weak assumptions.
- Run lightweight premortem.
- Identify failure paths.

Must not:

- Oppose for performance.
- Create fake risks without explaining likelihood or impact.

### 7.4 Operator

Purpose:

- Translate decisions into concrete actions.
- Identify execution cost and next steps.

### 7.5 Researcher

Purpose:

- Identify missing information.
- Separate known facts from assumptions.

MVP:

- Use mock research.
- Do not implement full web search unless explicitly available.

### 7.6 User Advocate

Purpose:

- Represent user's energy, values, emotional context, and lived constraints.
- Ensure the recommendation fits the human, not just the abstract optimization.

Must not:

- Diagnose the user.
- Create emotional dependency.

## 8. UI Requirements

The first screen should be the usable product, not a marketing landing page.

### 8.1 Main Layout

Use a three-zone layout:

- Left: session navigation and phase progress
- Center: council room and role messages
- Right: Decision Canvas
- Bottom: user input and meeting actions

On smaller screens, use tabs:

- Council
- Canvas
- Decision

### 8.2 Council Room

Represent each role with:

- Avatar or simple persona badge
- Role name
- Current state: idle, thinking, speaking, challenging, summarizing
- Last contribution summary

No need for complex animation in MVP.

### 8.3 Decision Canvas Panel

Show these sections:

- Problem
- Options
- Criteria
- Key claims
- Assumptions
- Risks
- Disagreements
- Recommendation
- Human decision
- Review plan

Sections can be collapsible.

### 8.4 User Meeting Actions

Provide buttons, not only a text box:

- Continue
- Ask a role
- Challenge this
- Clarify context
- Update criteria
- Generate recommendation
- Decide
- Defer
- Schedule review

## 9. Mock Agent Implementation

MVP may use deterministic mock agent outputs so the product can be built and tested without API keys.

Implement an interface:

```ts
interface AgentAdapter {
  run(input: AgentInput): Promise<StructuredAgentOutput>;
}
```

Implement:

```ts
class MockAgentAdapter implements AgentAdapter
class LlmAgentAdapter implements AgentAdapter // optional
```

Mock outputs should still be structured and role-specific.

## 10. Minimal Safety Behavior

Implement a simple risk classifier.

If user input appears to involve:

- self-harm
- violence
- medical diagnosis
- legal advice
- investment recommendation

Then do not proceed as a normal council decision.

Show safe support text and offer low-risk alternatives, such as:

- organize questions for a professional
- clarify non-professional considerations
- make a safety plan or seek immediate help for crisis cases

Do not implement clinical or regulated advice.

## 11. Storage

For MVP, one of these is acceptable:

- local JSON file
- browser localStorage
- SQLite

Persist:

- sessions
- canvas
- transcript
- decisions
- review plans

Do not persist sensitive long-term memory automatically in MVP.

## 12. Acceptance Criteria

The MVP is acceptable when:

1. User can create a decision session.
2. Session progresses through the defined phases.
3. At least six roles appear in the council room.
4. Each role can produce structured output.
5. Decision Canvas updates as the session progresses.
6. User can ask, challenge, clarify, decide, and defer.
7. User can create a final decision record.
8. User can schedule a review.
9. Data persists across reload.
10. High-risk input does not enter normal decision flow.
11. The app can run without real LLM API keys using mock mode.

## 14. Implementation Rules for Coding Agents

Do:

- Keep implementation focused on the MVP.
- Use structured data objects, not only chat messages.
- Keep user as final decision-maker.
- Make Decision Canvas central.
- Keep mock mode working.
- Leave extension points for real LLMs, web search, memory, and reliability engine.

Do not:

- Build a generic chatbot.
- Make a marketing landing page as the main product.
- Implement uncontrolled autonomous agents.
- Add regulated advice features.
- Hide important disagreement.
- Let majority vote decide automatically.
- Persist sensitive personal memory without explicit user approval.
